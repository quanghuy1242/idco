# Owned editor ‚Äî parity gaps to close (grounded against legacy Lexical + the reader plan)

## What this is

A backlog of what the owned-model editor (`packages/editor`) still misses versus the legacy Lexical editor (`packages/editor-legacy`) and the read tier (docs/015), found by grounding the two side by side. Each item names the gap, where it is specified, the current state, and ‚Äî for the SPI items ‚Äî the concept it should be built on, so it can be picked up one at a time. This is a working list, not a design doc; it carries enough concept to start each slice without re-deriving it. Layout-shell parity (field label, bordered frame, error slot) and the side/aside TOC rail are deliberately out of scope; the aside rail was removed by design and the shell is the host's concern.

## 1. Drag-to-reorder blocks

Legacy has `draggable-block-plugin.tsx` (grab a block by a handle, drop it elsewhere). The owned editor has no draggable / drop-indicator / move-by-drag ‚Äî it can `remove-block` but not reposition by drag (`view/render/object-block.tsx:92-94`). The one direct-manipulation affordance legacy had that the owned editor dropped. The model already supports moving nodes (the table column/row move composes `move-node`); this is the view-layer handle + drop-indicator + a move command, not new core.

## 2. Editor overlay SPI ‚Äî stop re-fixing React Aria popovers per call site

The recurring pain: every floating surface (selection flyout, cell `‚Ä¶` popover, ribbon `popover` actions, slash menu, link/annotation popovers) configures React Aria overlays raw, and each one re-derives the same focus/dismiss translation by hand ‚Äî usually in raw-div land with string-matched `data-engine-*` selectors. We keep patching the same class of bug: dismiss-on-blur when focus bounces to the editor (`selection-flyout.tsx shouldKeepFlyoutOpen`, `table-interactions.tsx keepCellPopoverOpen` + `pressInsideRef`), don't-blur-the-editor-on-press (`slash-menu.tsx onMouseDown preventDefault`, `ribbon.tsx onMouseDownCapture`), autofocus-because-non-modal-won't (`use-reveal-focus.ts useAutoFocusWithin`), and the `@idco/ui` popover's `isNonModal` + `shouldCloseOnInteractOutside` passthrough. Five symptoms, one missing abstraction.

Root cause, stated once: React Aria overlays assume DOM focus is authoritative (focus leaves ‚Üí dismiss; open ‚Üí grab focus; close ‚Üí restore). This editor assumes the opposite ‚Äî the model selection is authoritative and DOM focus serves it (EditContext host, `focusSelectionSoon`, selection survives blur). Every fix is the same translation patch on that mismatch, written again at each site. The awfulness is not React Aria; it is using a focus-authoritative overlay library directly against a focus-non-authoritative surface, per call site. Wrap it, do not drop it ‚Äî hand-rolling overlays would violate the UI philosophy and trade these bugs for worse a11y/positioning ones.

The idea: an editor-owned overlay adapter (an SPI in `packages/editor/src/view`, e.g. `view/overlay/`) that sits ON `@idco/ui`'s popover and encodes "focus is model-owned" once, so call sites declare intent, not React Aria mechanics. It must NOT live in `@idco/ui` ‚Äî the "editing surface owns focus" semantics are editor-specific and the product-neutral package must not learn about EditContext / `view-root` (the boundary lint would reject it). `@idco/ui` stays the React-Aria-behavior + DaisyUI-styling layer; the editor adds a thin focus-ownership policy layer above it. This is the missing sibling to the command-surface / node-view / side-panel / data-provider SPIs: an overlay SPI.

Three responsibilities, and resist growing past them (this is a small core hook, not an overlay framework): (1) Surface-aware dismissal, inverted ‚Äî the editor owns open/closed and React Aria becomes controlled and only *asks*; "RA never decides to close, it reports an interaction, the editor decides." The selection flyout already does this right (its coordinator owns it); the cell popover still leans on RA heuristics (the wrong model). Press-origin tracking + the surface-is-inside rule live here, defined against a real `surfaceRef`, not `closest("[data-engine-...]")`. (2) A declared focus policy ‚Äî probably one enum is the whole thing: `form` (takes focus, autofocuses a field ‚Äî glossary/comment/link) vs `actions` (focus-transparent, editor keeps focus, roving/keyboard ‚Äî slash menu, flyout toolbar, cell `‚Ä¶` menu). That distinction subsumes `useAutoFocusWithin`, the `preventDefault` guards, and the blur-dismiss patches, because the primitive then knows whether a focus move *to the editor* is expected (always, for `actions`) or a real dismissal. (3) One portal root ‚Äî portal every editor overlay into the editor root instead of `<body>`, so DOM ancestry matches React ancestry; containment checks become trivial and reliable, and the ribbon `onMouseDownCapture` hack (which exists only because React portals to body while synthetic events bubble the React tree) could likely be deleted.

Call-site shape should read as intent, e.g. `<EditorOverlay intent="actions" anchor={anchorRef} isOpen={open} onDismiss={‚Ä¶}>` ‚Äî no `shouldCloseOnInteractOutside`, no `isNonModal`, no `pressInsideRef`, no surface selectors at the call site. First real design decision when planning: #1 (invert dismissal ownership) ‚Äî get that contract right and the focus policy + portal fall out of it. Migration is incremental and self-justifying: move the flyout first (already ~80% this model), then cell popover, ribbon popovers, slash menu, link/annotation popovers ‚Äî each migration deletes a pile of ad-hoc guards, so it pays for itself as you go. The subtle part is `intent: actions` (focus-transparent yet keyboard-operable); the slash menu already proves it is doable, the SPI just makes it the default for menus.

Origin: surfaced while fixing two non-modal-popover bugs (cell color popover dismissing on swatch press / not dismissing on outside click; toolbar-opened glossary/comment inputs unable to mouse-select because the toolbar's capture-phase `preventDefault` reached them through the React tree). Those are now patched, but the patches are the evidence this abstraction is missing.

## 3. Duplication the `check:dup` gate is built not to notice

The fear is "vibe-coded `as any` / `isRecord` everywhere." Measured, that fear is misaimed: `as any` appears once across editor/reader/ui, `@ts-ignore`/`@ts-expect-error` zero ‚Äî the type discipline is actually tight. The real debt is the opposite shape: the same *logic* maintained in parallel across packages, which is exactly what a duplicate gate should catch and this one structurally cannot. `scripts/check-duplication-threshold.mjs` is a repo-wide percentage gate (fail only above 3% mild-mode density); the repo sits at 1.9% (1713 duplicated lines, 37 clone groups), green. A percentage gate measures aggregate density, not per-block badness, so a single 350-line verbatim mirror never moves the needle and sails through. The semantic detector that *does* surface conceptual dupes (`pnpm advise:dupes`, fallow `--mode semantic`) exists but is not wired into `pnpm check`, so nothing fails on it.

Cluster A ‚Äî the editor‚Üîreader model mirror (`packages/editor/src/core/model` ‚Üî `packages/reader/src/reader/model`). `segmentText` is near-verbatim duplicated: same boundary-set construction, same sticky-merge branch, same sort, only the type names differ (`ResolvedMark`‚Üí`ReaderResolvedMark`). `marks.ts:74-115` ‚â° `reader/model.ts:195-229` (337 tokens), plus model.ts‚Üîmodel.ts, url-safety, and mark-render pairs, and the parity tests are themselves duplicated (`tests/editor/reader-resting-parity.test.tsx` ‚Üî `tests/reader.test.tsx`). This is the documented [[reader-convergence-028]] decision ‚Äî resolution mirrored-and-parity-tested rather than relocated ‚Äî so it is intentional, but it is still live duplication that drifts silently if one copy is edited and the parity test does not cover the edited path. Action is not "dedupe blindly"; it is "make the mirror a *declared* exception" so the gate acknowledges it instead of passing in ignorance.

Cluster B ‚Äî two parallel rich-text renderers (`packages/reader/src/l1/*` ‚Üî `packages/ui/src/rich-text-content.tsx`), nine clone groups. `verticalAlignClass` and `readableTextColor` are literally copy-pasted ‚Äî the reader copy's own comment reads "Moved verbatim from `@idco/ui` rich-text-content" ‚Äî alongside table rendering (169+147 tokens), toc, objects, and types. This is the "reader / retire content-renderer" backlog item ([[editor-spi-gap-epic]]) quantified: ~900 tokens of a renderer that is supposed to go away but is being maintained twice in the meantime. Retiring `rich-text-content.tsx` in favor of the reader path collapses this cluster wholesale; until then it is the single largest avoidable dup surface in the tree.

Cluster C ‚Äî `isRecord`, the micro-dup no token detector can see. Defined identically twice (`packages/editor/src/core/registry/object-registry.ts:595`, `packages/lib/src/auth-fetch.ts:160`) and reimplemented inline as a `typeof x === "object"` guard in ~13 files across editor/reader/lib. Each guard is ~6 tokens, below fallow's 50-token floor, so it is invisible to *any* token-based detector ‚Äî the only thing that would catch it is a lint rule ("no inline record guard; import the shared one"), which does not exist. This is the class the user actually feels, and it is a lint problem, not a dup-gate problem.

Recommendations, smallest-leverage first. (1) Wire `advise:dupes` (semantic mode) into `pnpm check` as an allowlist gate: any *new* cross-package clone group fails; the two known-intentional mirrors (Cluster A model + its parity tests) sit in an explicit allowlist file, so the deliberate exception is recorded and a *third* mirror trips the wire. This converts "passes because the percent is low" into "passes because each dup is named." (2) Add one architecture-lint rule banning inline record guards, pointing at a single shared `isRecord` (promote one of the two definitions into `packages/lib` and import it) ‚Äî this is the only thing that catches Cluster C at all. (3) Treat Cluster B as the existing retire-content-renderer backlog item, not new work; closing that item deletes nine clone groups for free. (4) Separately, `pnpm advise:aislop` already flags 7 errors worth a triage pass independent of duplication: a possible secret in `stories/engine-owned-model.stories.tsx:36`, two `dangerouslySetInnerHTML` sites (`reader/l1/blocks.tsx:217`, `ui/code-editor.tsx:135`), and `hono`/`undici` advisories. The throughline: the gate that exists rewards aggregate cleanliness; the duplication that hurts is concentrated and cross-package, so the fix is per-block accountability (allowlist + one lint rule), not a lower percentage threshold.

**Resolution (2026-06-25) ‚Äî recommendations (1), (2), and the dedupable parts of (3) shipped.** (1) `scripts/check-semantic-dupes.mjs` runs the exact `advise:dupes` semantic detection and holds it against `scripts/semantic-dupes-allowlist.json`; it is wired into `pnpm check` as `check:dup:semantic` (alongside, not replacing, the percentage gate). Each clone *family* is keyed by its file-set (stable across line moves) and carries a content fingerprint, so a new cross-package clone fails for lacking an entry, an allowlisted mirror that silently drifts fails on fingerprint mismatch, and an entry with no written reason fails ‚Äî `node scripts/check-semantic-dupes.mjs --update` refreshes fingerprints and surfaces new families without ever blessing one reasonlessly. The six surviving families are each named with a reason: Cluster A's `segmentText` mirror (docs/028), the two engine‚Üîeditor-legacy reimplementations (perf-metrics + column geometry, intentional because engine core cannot import the Lexical legacy package), the editor-barrel‚Üînav-icons detector false positive, and Cluster B's two renderer mirrors. (2) `architecture/no-inline-record-guard` (in `scripts/oxlint-js-plugins/architecture.js`, wired in `.oxlintrc.json`) bans inline `typeof x === "object"` record guards in @idco package source; the canonical `isRecord` now lives in `packages/lib/src/guards.ts` and all three former definitions + ~12 inline guards across editor/editor-legacy/reader import it. (3 partial) The pure copy-pasted helpers `verticalAlignClass`/`readableTextColor` moved to `packages/lib/src/rich-text-style.ts`, deleting one Cluster B clone family (reader/types ‚Üî ui rich-text-content) outright; the irreducible JSX renderer mirrors (table/toc) stay allowlisted against the retire-content-renderer backlog, which the cross-RSC-boundary constraint (reader L1 may not import @idco/ui) keeps as genuinely separate work. **Still open:** retiring `rich-text-content.tsx` itself (closes the last two Cluster B families and is a consumer-spanning migration), and recommendation (4)'s aislop security triage, which the note already brackets as independent of duplication.

## 4. Markdown / lists / export / publication hardening (2026-06-25 audit)

A grounded backlog across five areas the author asked about. The underlying *nodes* mostly already exist; these gaps are in the input, transport, and output layers around them. Block nodes present today: `heading` (h1‚Äìh6 via `tag`), `paragraph`, `listitem`, `quote`, `callout`, `code-block`, `media`, `embed`, `toc`, `table`, `post-ref`, `divider` (HR) ‚Äî `core/registry/object-registry.ts:144-335`, `core/model/model.ts:200-215`. Inline marks present: `bold`, `italic`, `underline`, `strikethrough`, `code`, `highlight`, `subscript`, `superscript`, `link` ‚Äî `core/model/model.ts:172-184`. The document snapshot already carries a top-level `settings: DocumentSettings` that round-trips through compat (`model.ts:265,285`), but it is an untyped `JsonObject`. So most items below are about *reaching* existing nodes (by typing, paste, export) and *typing* the settings bag ‚Äî not new block types. **Today's focus = ¬ß4.1, ¬ß4.2, ¬ß4.3. ¬ß4.4 export and ¬ß4.5 page settings are deferred.** Calibration up front: not all of these are "easy" ‚Äî the truly trivial ones (h4‚Äìh6 prefixes, persist list indent) sit beside two items that carry a real decision (the markdown-paste parser is a from-scratch parser; true list nesting is the synthesize-vs-structural fork in ¬ß4.3a). The rest is mechanical once those two calls are made.

### 4.1 Markdown typing affordances are shallow (`core/markdown-shortcuts.ts`) ‚Äî TODAY

Current state: the detector is single-shot. Block prefixes fire only when the caret sits exactly after the prefix at offset 0 (`markdown-shortcuts.ts:117-129`); the only inline affordances are inline-code via a closing backtick, smart quotes, and bracket auto-pairing (`markdown-shortcuts.ts:142-162`). The view runs it once per single-character input in `view/render/text-block.tsx:174-180`. No multi-character paired-marker detection, no fence handling, no snippet/cascade engine.

Missing block prefixes: headings h4‚Äìh6 (`#### `‚Ä¶, only h1‚Äìh3 exist at `:88-90`; trivial, `heading` already takes any `tag`); task/checklist `- [ ] ` / `- [x] ` (the `listitem` `checked` attr exists at `model.ts:315` but nothing creates it ‚Äî see ¬ß4.3c); horizontal rule `---`/`***`/`___` on its own line ‚Üí insert a `divider` object (exists, `object-registry.ts:324`); code fence ` ``` ` ‚Üí open a `code-block` object (exists, `:335`) capturing the language token. The HR and fence cases are *line‚Üíobject* shortcuts ‚Äî a shape the current detector (which only retypes a `TextLeafType`) does not model; they need a new `BlockShortcut` variant compiling to an object insert, and the fence additionally redirects subsequent input into the new block (the biggest block item). Also: only `1. ` is recognized; an arbitrary ordered start (`2. `, `7. `) needs a `value`/start attr or the adjacency-based ordinal paint (docs/018 ¬ß2.10) silently renumbers it.

Missing inline (paired-marker) affordances: bold `**`/`__`, italic `*`/`_`, strikethrough `~~`, highlight `==`, inline link `[text](url)`, autolink of a bare URL ‚Äî all the marks exist and are reachable only by shortcut/toolbar today. Architecture play: inline-code already does the "scan back from the just-typed closing marker to a matching opener on the same line" move (`:144-150`); generalize *that one mechanism* into a paired-marker table (`marker ‚Üí mark kind`, multi-char open/close, min-run guard) rather than six bespoke branches ‚Äî one detector covers bold/italic/strike/highlight/code/link. Restraint applies (`[[spi-generalization-restraint]]`): this is the small core hook, not an autoformat plugin system, and explicitly **not** a multi-keystroke cascade/snippet engine (high-risk on the live-EditContext path, no concrete need ‚Äî record that as a closed decision so it stops resurfacing).

### 4.2 Markdown paste + clipboard fidelity ‚Äî TODAY

No markdown-on-paste: `use-clipboard.ts:117-120` reads `text/plain` and inserts it verbatim via `insert-text`; `text/markdown` is never read; the ¬ß4.1 block prefixes only fire on live single-char input so they do not help paste (`text-block.tsx:174`). Pasting a `.md` document drops all structure. Proposed: a pure markdown‚Üísnapshot-fragment parser (block + inline) that paste routes `text/markdown` (and optionally heuristically-markdown `text/plain`) through, producing the same node fragment the HTML-paste path builds, inserted as a multi-block paste. This is a from-scratch parser ‚Äî the least "easy" item here; keep it pure/testable like the detector, and scope it to the representable node set rather than full CommonMark on day one.

HTML-paste gaps (`view/paste-html.ts`): paste maps headings/paragraphs/quote/list-items/code/links + inline formats, but drops tables (not in `BLOCK_TAGS` ‚Üí degrade to cell text), images/`figure`/`figcaption` (not imported), and list nesting (flattened). Closing these reaches nodes that already exist (`table`, `media`) and shares the fragment builder with the ¬ß4.2 markdown parser ‚Äî build it once. Clipboard image/file paste: `use-clipboard.ts` never inspects `clipboardData` for `image/*`; only drag-drop of images works (`owned-model-editor.tsx:180-207`) ‚Äî add clipboard-image paste reusing that drop‚Üí`uploadImage`‚Üí`media` path. Copy/cut write `text/plain` only (`use-clipboard.ts:62,76`, via `collectSelectionText` in `core/store/selection.ts:61-101`), so editor‚Üíeditor paste downgrades to plain text ‚Äî also write `text/html` and (once ¬ß4.4 lands) `text/markdown`, and prefer the editor's own payload on internal paste. That last part depends on ¬ß4.4's serializers; do export first or share its block-walker.

### 4.3 Lists: nesting, persistence, checklist, styles ‚Äî TODAY (`view/nodes/list.tsx`, `core/commands/blocks.ts`)

Lists are flat-by-design: `listitem` is a `TextLeafType` and depth is an `attrs.indent` integer rendered as left margin, not structural containment (docs/018 ¬ß2.10; `flat-blocks.ts:256`). A real structural `list` node and a `compileIndentItem` nesting path exist but are dead-by-design (`blocks.ts:374-457`, annotated UNREACHABLE).

(a) **True sublists vs. flat indent ‚Äî a decision, not just code.** Today indent is purely visual; an indented bullet is still a ROOT child, so semantic `<ul><ol>` nesting, nested-list ordinal numbering, and markdown/HTML export of nested lists are all impossible from flat indent. Two roads: (i) keep flat leaves and *synthesize* the nesting tree at render/serialize time from the `indent` attr (cheaper, preserves per-item virtualization, but every consumer reconstructs the tree), or (ii) revive the structural `list` path (truthful tree, but reopens the recursive sub-block windowing the flat design deliberately deferred). This is the first real design decision in this section and gates 4.3b/4.3d and the list parts of ¬ß4.2/¬ß4.4. Recommend (i) synthesize-from-indent first ‚Äî unblocks export and HTML paste without reopening virtualization ‚Äî and treat (ii) as the eventual multi-block-container work. (b) **Indent is dropped on save ‚Äî a real data-loss bug, not just a gap.** `attrs.indent` is editor-session-only and is not carried through the compat export (docs/018 ¬ß2.8), so even the visual nesting an author makes today vanishes on reload. Highest-value, lowest-risk item: persist `indent` through compat import/export; independent of the 4.3a decision. (c) **Checklist has no UI.** The `checked` attr exists (`model.ts:315`) and legacy had full checklist UI, but import maps check lists to bullet (`payload-import.ts:188`) and there is no command/markdown to create or toggle one ‚Äî add the `- [ ] ` prefix (¬ß4.1), a toolbar/slash command, and a clickable checkbox in `list.tsx`. (d) **No glyph/numbering-style variants.** Bullets are a hardcoded `‚Ä¢`, ordered lists a hardcoded decimal (`list.tsx:26-42`, `view/styles.ts`); no model field for disc/circle/square/dash or 1./a./i./A./I. ‚Äî needs a `listStyle` attr, render mapping, serialize support. Lowest priority; do after a‚Äìc.

### 4.4 Export: markdown + custom syntax ‚Äî LATER (new; nothing exists)

No `toMarkdown`/`toHTML`/any export serializer exists in editor or reader; outputs are the native `EditorDocumentSnapshot` JSON and the legacy `RichTextCompatDocument` round-trip shape (`compat/compat.ts:198-217`). Proposed shape: a pure snapshot‚Üítext serializer walking `body.order`/`blocks`, with a per-object-kind `serialize(node) ‚Üí string` seam on the node definition (`object-registry.ts`) so each object owns its custom syntax ‚Äî callouts as `:::note` directives, code/mermaid as fenced blocks, divider as `---`, media as image syntax. Two hard constraints from docs/006: objects export their **baked static** representation only, never a live/computed one (006 ¬ß5.8); and markdown export must compose with the ¬ß4.2 parser so editor‚Üímd‚Üíeditor is lossless for representable nodes, with a documented lossy set. Define the custom directive grammar once, here, as the export contract. Also unblocks the ¬ß4.2 copy-as-markdown fidelity item.

### 4.5 Page / publication settings ‚Äî LATER (docs/006 ¬ß6)

The contract slot exists ‚Äî `settings: DocumentSettings` round-trips (`model.ts:265,285`; `compat.ts:178,215`) ‚Äî but it is an untyped `JsonObject`: no publication schema, no UI surface, no renderer parity. docs/006 ¬ß6.2 sketches the target `RichTextPublicationSettings` (page size, orientation, margins, header/footer, page numbers) and ¬ß6.5 the body-vs-settings boundary (global layout = settings; manual page-break = body node). Three pieces in order: (a) a *typed* `settings.publication` shape under the existing open bag ‚Äî additive, the bag already passes unknown fields through (`compat.ts:176-180`); (b) a View-tab/object-chrome surface to edit it (006 ¬ß4.5) on React Aria + DaisyUI; (c) renderer parity in `@idco/reader` honoring the same settings with a host override (006 ¬ß6.3). Open upstream decision (006 ¬ß6.4): reflowable vs fixed-layout output determines how much of this is load-bearing vs advisory ‚Äî settle with the host before building the fixed-layout pagination stage (its own large workstream). Editor's obligation regardless: preserve settings it accepts, never strip them.

Cross-cutting for all of ¬ß4: every markdown/list/export feature must round-trip editor‚Üîreader‚Üîexport and carry a parity test (`[[reader-convergence-028]]`), and none of it pulls product/runtime deps into the editor (shared-package boundary).

### 5.1 B1 ‚Äî [BUG/PKG] bake Web Worker referenced as `.ts`, only `.js` is shipped ‚Üí consumer build fails

**Resolution (2026-06-28) ‚Äî fixed + guarded.** The editor build runs `scripts/fix-worker-specifier.mjs .` after `tsc` (before `gen-api-map`): it rewrites every `new URL("‚Ä¶.ts", import.meta.url)` worker specifier in `dist/**/*.js` to `.js`, so the shipped `dist/view/react-view.js` now constructs the worker from `bake.worker.js` (which exists in dist). Source keeps `.ts` so Vite/Ladle dev still resolves against `src`. Guarded by `scripts/check-package-contract.mjs` (`pnpm check:package`, wired into `pnpm check` after build): it scans dist for `new Worker(new URL(...))` and fails if any worker URL resolves to a file not in dist ‚Äî proven RED (reverting the dist specifier to `.ts`) ‚Üí GREEN. Consumer-facing, so content-api gets it after a republish. Note: the dist also uses extensionless/directory imports (`./view` ‚Üí `./view/index.js`) that Node's native ESM resolver rejects but bundlers (Vite/webpack/esbuild) resolve ‚Äî expected for a bundler-targeted package, not a bug.

### 5.3 B3 ‚Äî [BUG, virtualized-path] caret lost after a structural edit under virtualization ‚Äî now isolated to idco; the empty-doc no-caret symptom is probably a different issue

Original status from content-api: filed `[UNVERIFIED]` ‚Äî observed downstream, not yet isolated to idco vs the content-api integration. Three reported symptoms (content-api, before isolation): (1) an empty document (single empty paragraph) shows **no caret** when focused; (2) **insert a table, then delete it ‚Üí the caret is completely lost** and does not return without re-clicking; (3) the content area is unusually tall with empty space below short content. Known at filing time: content-api mounted with the engine default `virtualize = true` (`react-view.tsx:176`); the idco stories all use `virtualize={false}`. The report prescribed a two-step isolation repro (content-api with `virtualize={false}`, then an idco story with `virtualize={true}`) and said explicitly: do not change idco code until it confirms.

Our isolation (2026-06-28) ‚Äî read the virtualized caret/overlay/focus path; verdict: **symptom #2 is a genuine idco virtualized-path bug (content-api exonerated); symptom #1 is probably not B3 at all.** The new data point the report was waiting for is in hand: the author confirms the caret vanishes specifically with `virtualize={true}`.

Mechanism. The painted caret has two gates in `packages/editor/src/view/overlays/selection-overlay.tsx`, and both must hold or it paints nothing. Gate 1 ‚Äî the focus block must be mounted: the collapsed-caret branch only emits a rect when `blockRefs.get(focus.node)` exists (`selection-overlay.tsx:85-98`); an offscreen/unmounted block paints nothing (docs/011 ¬ß8.5). Gate 2 ‚Äî DOM focus must be inside the surface: `useEditorFocusWithin` returns `within && doc.hasFocus()` where `within = activeElement === root || root.contains(activeElement)` (`selection-overlay.tsx:383-411`); when it is false, `SelectionOverlay` **filters out every `caret` and `gap` rect** (`selection-overlay.tsx:429-431`) ‚Äî range highlights survive, the caret/gap do not. That filter is the "caret completely lost, does not return without re-clicking" symptom.

Why insert-then-delete-table loses the caret only under virtualization. A structural edit mutates `store.order`, which bumps `reseedVersion` (`packages/editor/src/view/controllers/use-virtual-window.ts:79-82,152-156`, "force a bulk re-seed of unmounted blocks when a document-wide [change]") ‚Üí blocks remount as fresh DOM nodes, and a fresh node does not inherit DOM focus. The re-homing mechanism is `syncFocusToSelection` (`packages/editor/src/view/controllers/use-focus-navigation.ts:337-356`), called by every structural/clipboard/touch edit path (use-clipboard, use-gap-cursor, use-touch-selection, react-view), but it tries focus once now and then **retries on exactly one `requestFrame`**. The reseed remount (a React state update, plus font-settle / ResizeObserver measurement bumps that re-render later) can land on a *different* frame than that single retry, so `focusBlock` finds a detached or not-yet-mounted ref, focus never lands, gate 2 stays false, and the caret is filtered. The non-virtualized path never remounts the surviving block (all blocks stay mounted, same element), so DOM focus rides through the edit and the caret stays ‚Äî that asymmetry is exactly why it only reproduces with `virtualize={true}`, and is what pins the bug to idco rather than content-api.

Why symptom #1 (empty-doc no caret) is likely a different animal. For a one-block empty doc, block 0 is in-window and mounted (gate 1 passes), and the caret falls back to the block's bounding rect (`selection-overlay.tsx:297`, `fallbackCaretRect` at `:322-343`), so it *should* paint if focused. If it does not, it is gate 2: focus never actually landed on the empty block (click ‚Üí `placeCaretAt` ‚Üí `resolveTextPointAt` can fail to resolve a text point in an empty block / the tall empty area). That path is shared with `virtualize={false}` and is entangled with **D3** (no real caret seed) and **R2** (placeholder). Prediction to confirm: symptom #1 reproduces even with virtualization off, which would make it the empty-doc seed/focus issue, not B3.

Repro plan (agreed: do the e2e first). No existing story combines `virtualize={true}` with a small/empty doc, so the decisive test needs a throwaway virtualized small-doc story (mirroring `Phase8ToolbarEditor` but virtualized) plus a Playwright spec under `tests/e2e/` that (a) loads the empty-doc virtualized story, focuses, asserts `[data-engine-caret]` exists ‚Äî to settle symptom #1 separately; (b) loads the small-doc virtualized story, inserts a table then deletes it (toolbar, or the diagnostics global `ENGINE_VIEW_API_KEY` / `__IDCO_ENGINE_*` store command since `OwnedModelStoryFrame` has no toolbar), and asserts `[data-engine-caret]` survives ‚Äî to confirm symptom #2. The existing harness is the model: `tests/e2e/engine-caret.spec.ts` loads `/?story=<id>` and queries `[data-engine-caret]` / `[data-engine-block-id]` / `[data-engine-view-root]`.

Fix direction (after the repro, P1 once confirmed). Stop relying on a single deferred frame: re-home focus at **mount time** ‚Äî when a block element mounts and the model selection names a text focus on it but nothing in the surface holds DOM focus, claim focus then (it cannot lose the race). Alternatives: keep the survivor block's element identity stable across a reseed; or make gate 2 treat "model text-focus on a mounted block, DOM focus momentarily on `body` during a remount this frame" as still-focused and re-assert. Relates to the focus-reclaim seam ([[editor-overlay-authority-029]], `editor-store.ts:isReclaimSuspended`) and the B1/B‚Ä≤ mobile-keyboard focus work ([[owned-editor-mobile-keyboard-flicker]]).

**Resolution (2026-06-28) ‚Äî reproduced with Playwright and fixed; the mechanism was simpler than the pre-repro theory, and symptom #1 was not a bug.** Two repro stories (`B3VirtualizedSmallDoc` = virtualized small doc + toolbar; `B3VirtualizedEmptyDoc` = virtualized single empty paragraph) and `tests/e2e/engine-b3-virtualized-caret.spec.ts` drive it. What the repro actually showed, correcting the theory above: (a) it is **not** the `reseedVersion`/remount race ‚Äî a structural edit reconciles the offset model in place and blocks are keyed by id, so the survivor never remounts; (b) symptom #1 does **not** reproduce ‚Äî an empty virtualized doc paints a caret fine (so the empty-doc caret is a non-bug; D3's `emptyDocument()` factory is still wanted as DX, but the caret itself works once a paragraph is seeded); (c) the real symptom #2 mechanism: deleting the table the caret sat in (caret was inside a table cell, a nested sub-engine host) removes that host, and removing a focused element drops browser focus to `<body>` ‚Äî confirmed by a focus-event probe: only a capture-phase `blur` fires on the cell host with `isConnected` still `true` and `document.activeElement` already `body`, and **no** `focusout` reaches the document. The model correctly remaps the caret onto a surviving block (commit pipeline `apply‚ÜímapSelection`), but nothing re-homes DOM focus, so the focus-within gate (`selection-overlay.tsx:429-431`) filters the painted caret and typing is dead until a re-click. The toolbar's own `focusEditor` masks it; a bare `store.command({type:"remove-block"})` (a host button, a programmatic delete ‚Äî content-api's path) does not.

Fix shipped: a **commit-driven focus reclaim** in `react-view.tsx` (the `store.subscribeCommit` effect). On a **local** `structureChanged` commit, defer one frame (focus settles on body the same way the focus-within gate defers) and, only if a focus-taking overlay is not deliberately holding focus (`store.isReclaimSuspended()`, the docs/029 ¬ß7.1 reclaim seam), the model selection is a text caret, focus actually fell to `body`/`documentElement`, and that caret's block is mounted, call `syncFocusToSelection()` (DOM focus only ‚Äî it never mutates the model). The trigger is the commit, not a focus event, because that is the only reliable signal that focus was orphaned **by an edit** rather than by a deliberate user blur (which must still hide the caret). Excluded for free: virtualization scroll-away (the focus block is then unmounted, so the `blockRefs.has` guard fails and nothing is reclaimed) and remote/collaborative edits (`origin !== "local"`). Proven RED before / GREEN after; the empty-doc guard also passes. `pnpm check` green. Caveat: the broader engine e2e suite has ~6 specs that fail on this WSL host (headless-Chromium scroll/clipboard/popover timing) **identically with and without the fix** ‚Äî confirmed pre-existing/environmental, not a regression. Consumer-facing, so content-api gets it only after a republish.

### 5.4 D1 ‚Äî [DX] `@idco/editor` is not import-safe under SSR

Symptom (severity: medium): importing `@idco/editor` into a server/SSR module graph (a Cloudflare Worker SSR pass) crashes the render ‚Äî every admin page that transitively imports the editor binding hit the server error boundary. Root cause: the package reaches for DOM/EditContext globals at module-load time; it is a client-only engine but nothing signals or guards it. content-api workaround: the editor is loaded only via `next/dynamic(() => import("./record-editor-impl"), { ssr: false })` behind an SSR-safe wrapper that imports the editor as a type only (`React.lazy` was insufficient ‚Äî the bundler still evaluated the module during SSR). Suggested fix: guard module-load-time browser access so a bare `import "@idco/editor"` does not throw on the server (defer DOM access to first render/mount), or document prominently that the package is client-only and must be dynamically imported with SSR disabled.

Our refinement (confirmed behavior, P1): client-only is **by design** ‚Äî the reader is the RSC-safe tier and the dependency direction is editor ‚Üí reader-L1, never reverse ([[reader-convergence-028]], [[owned-model-editor-design]]). So the right framing is "guard + document," not "make the editor isomorphic." Pragmatic first step: a clear client-only guard/error at the entry plus one prominent doc line on the dynamic-import-with-`ssr:false` requirement; a full defer-DOM-to-mount refactor is larger and only worth it if a consumer needs a bare import. The release smoke test (¬ß5 root cause) should include an SSR module-eval assertion so a regression is caught.

**Resolution (2026-06-28) ‚Äî proven a non-bug; guarded + documented, no code fix.** Reproduced the SSR path faithfully: importing all four barrels (`editor`/`reader`/`ui`/`lib`) in a real DOM-less node environment (asserted `typeof window === "undefined"` and `typeof document === "undefined"`) does **not** throw ‚Äî the packages are import-safe at module load (no top-level DOM/EditContext access; the grepped DOM uses are all inside functions/effects). So D1's "a bare import crashes the server" does not reproduce ‚Äî like B3 #1, the reported framing was a non-bug. content-api's SSR crash was at *render* (a client-only interactive component rendered on the server), which is inherent to any client-only component and correctly handled by their `next/dynamic(..., { ssr: false })` ‚Äî that is the right pattern, not an idco code change. The earlier `node import` "Directory import not supported" error was just Node's native ESM resolver rejecting the dist directory imports (`./view`); bundler consumers (vinext/Vite/webpack) resolve those, so it is not a consumer bug. Deliverables: a permanent regression guard ‚Äî `tests/ssr-import-safety.test.ts` run in a node environment via `vitest.ssr.config.ts` (`pnpm test:ssr`, wired into `pnpm check`) so any future module-load DOM access fails the gate; the workspace src aliases were extracted to `vitest.shared.ts` so the two vitest configs share one list. The client-only / `ssr:false` contract is documented in docs/033 ¬ß3. A friendlier dev-time "rendered on the server" error was considered and left out (the `ssr:false` pattern + doc is sufficient; a guard that throws on server render would need to distinguish SSR from a legitimate test renderer).

### 5.5 D2 ‚Äî [DX] `CodeEditor` depends on a pre-existing global `Prism`

Symptom (severity: medium): rendering a code block throws `ReferenceError: Prism is not defined` unless the host already set up a global `Prism`. Root cause (as filed): `@idco/ui`'s `CodeEditor` loads PrismJS language components that register onto a global `Prism`, but the package does not establish that global itself. content-api workaround: a `prism-setup` module sets `globalThis.Prism` and is imported (via the dynamic loader) before the editor chunk. Suggested fix: have `CodeEditor` own its Prism instance (or initialize the global before loading language components); ship the `.token.*` palette as part of the B2 CSS entry.

Our refinement (verified, P1): the nuance matters for the fix. `packages/ui/src/code-editor.tsx:4` already does `import Prism from "prismjs"` and `:7-11` import the grammar packs (`prism-json`/`prism-typescript`/`prism-jsx`/`prism-tsx`/`prism-python`). The actual bug is that those grammar component files reference the **bare global `Prism`**, which `ReferenceError`s in a bundled ESM context unless `globalThis.Prism` is set *before* they evaluate. So "CodeEditor needs a host global" is accurate, but the fix is import-ordering inside `code-editor.tsx`: set `globalThis.Prism = Prism` before the grammar side-effect imports run (subtle, because ESM side-effect imports hoist ‚Äî likely needs the grammar imports pulled into a small module imported after the global is assigned). ### 5.8 R2 ‚Äî [REQUEST] `OwnedModelEditor` empty-document placeholder ("Type here‚Ä¶")

What: a `placeholder` prop rendering a muted hint in an empty document, like a normal editor's empty state. Why: a borderless full-page editor that is visually empty gives the author no signal it is editable ‚Äî every comparable editor (Notion, Docs, the GitHub composer) shows a placeholder in the first empty block. ### 5.9 R3 ‚Äî [REQUEST] chromeless / fill-height editor surface + click-empty‚Üícaret-at-end (with the reflow-vs-page-settings clarification)

What: first-class props to (a) drop the editor's default bordered card and (b) make the surface fill its container's height, with **clicking anywhere in the empty area placing a real text caret at the end of the document**. Why: for a document-centric CMS the editor *is* the page ‚Äî no card border, and the whole tall area is a click-to-type target (click below the last paragraph ‚Üí caret at end, like Word). Today the editable area is only as tall as its content, so the empty space below the text is dead. Current state (verified): content-api passes `style={{ border: "none", borderRadius: 0, maxWidth: "none", minHeight: "70vh" }}` and the engine merges `style` over `baseViewStyle` (`react-view.tsx:478` non-virtualized, `:532` virtualized), so this removes the border and stands the surface tall ‚Äî appearance works, but it relies on an inline-`style` escape hatch where a semantic prop belongs, `minHeight` does not truly *fill* a flex column, and it does **not** guarantee the click-empty‚Üícaret-at-end behavior (`onRootMouseDown` / `useGapCursor` handles empty-space clicks but is not clearly the "caret at end of doc" affordance a tall blank page needs). Suggested API: `chromeless?: boolean` (no border/radius/maxWidth cap); `fillHeight?: boolean` (or `minHeight?`); and the click-anywhere-resolves-to-nearest-text-position-biased-to-doc-end behavior.

Our refinement (verified, P2). The appearance half is genuinely reachable today via the `style` merge (confirmed at `react-view.tsx:478,532`); the substantive gap is the click-empty‚Üí**text caret at doc end** behavior, not a gap cursor. On height specifically: there *is* a `viewportHeight?: number` prop (`react-view.tsx:131-132`) but it is a **fixed pixel** scroller height applied only on the virtualized path (`:533`) and explicitly ignored when `virtualize={false}` ‚Äî which is the path content-api uses, so for them there is no height prop at all, only the `style` hatch. The clean fix is to **generalize `viewportHeight` into one typed height concept across both paths** (a number for fixed, or a `fill`/`100%` mode that stretches the container) rather than bolt on a separate `fillHeight`, and to expose `chromeless` as a typed prop so the `baseViewStyle` override stops being load-bearing consumer knowledge.

Reflow vs page-settings clarification (the author's concern that this "overlaps with page settings, docs/006, we are doing reflow"): it is a naming collision, not a functional dependency. docs/006 has two distinct "page" concepts ‚Äî (1) ¬ß6.1/¬ß6.4 **publication settings** (A4/A5/Letter, margins, running headers/footers, page numbers), which docs/006 puts in the **fixed-layout output target**, calls "effectively its own rendering engine," and records the reflowable-vs-fixed-layout choice as an explicitly **deferred open decision** (this is also ¬ß4.5 above); and (2) what R3 wants ‚Äî a chromeless, full-height **editing surface**, which is pure surface framing, still reflow, still a flowing article (the Word/Notion *feel*, not Word's *paged* model). So full-height chromeless editing is reflow + CSS + a caret affordance and does **not** require touching page size / margins / pagination; those stay in the deferred ¬ß4.5 / docs/006 fixed-layout workstream. The only thing literal A4-page-canvas editing would pull in is that fixed-layout stage ‚Äî out of scope for R3.

**Resolution (2026-06-28) ‚Äî FIXED, all three halves.** Two typed props on `OwnedModelEditorViewProps` (threaded through `OwnedModelEditor`): `chromeless?: boolean` and `fillHeight?: boolean`. Both inline style objects in `react-view.tsx` were folded into one `resolveViewStyle()` helper (`view/styles.ts`): `chromeless` strips the card chrome (border, radius, max-width cap) keeping prose font/colour ‚Äî the typed form of content-api's `style={{ border:"none", ... }}` override; `fillHeight` stretches the surface (virtualized scroller ‚Üí `height:100%`, non-virtualized ‚Üí `minHeight:100%` so it fills yet still grows past the viewport); the caller's explicit `style` still spreads last (back-compat escape hatch preserved). Per the refinement, `viewportHeight` was generalized rather than duplicated: on the virtualized path `useVirtualWindow` now measures the flex container with a `ResizeObserver` and windows against that measured height when `fillHeight` is set (falling back to `viewportHeight` until measured), so one height concept drives both paths. The composed `OwnedModelEditor` also stretches its chrome flex chain (wrapper‚Üíbody‚Üísurface‚Üíprose all `flex-1 min-h-0`/`h-full`) when `fillHeight`, so the view has a definite height to fill. The substantive **click-empty‚Üícaret-at-end** half: `useGapCursor.onRootMouseDown` now, for a body click below every mounted block (new `docEndPointBelowContent` helper), places a real text caret at the END of the last text leaf ‚Äî the Word/Notion "click the blank page under the text, type where it ends" affordance ‚Äî guarded to only fire below content (clicks between/over blocks keep the nearest-text mapping) and only when the last leaf is mounted (never focuses a windowed-out block). Stories `R3 chromeless fill height` (non-virtualized = content-api's path), `R3 fill height virtualized` (exercises the measured viewport), `R3 chromeless empty bare view`; unit tests for the `resolveViewStyle` matrix + e2e `engine-r-backlog.spec.ts` asserts the surface stretches and a blank-area click lands the caret at doc end (cross-browser chromium/webkit/firefox); screenshots `r3-chromeless-fill.png` / `r3-fill-virtualized.png` / `r3-bare-empty.png`. Orthogonal to the ¬ß4.5 / docs/006 page-settings workstream as analyzed above. Needs republish.
