# Owned editor — parity gaps to close (grounded against legacy Lexical + the reader plan)

## What this is

A backlog of what the owned-model editor (`packages/editor`) still misses versus the legacy Lexical editor (`packages/editor-legacy`) and the read tier (docs/015), found by grounding the two side by side. Each item names the gap, where it is specified, the current state, and — for the SPI items — the concept it should be built on, so it can be picked up one at a time. This is a working list, not a design doc; it carries enough concept to start each slice without re-deriving it. Layout-shell parity (field label, bordered frame, error slot) and the side/aside TOC rail are deliberately out of scope; the aside rail was removed by design and the shell is the host's concern.

## 1. Drag-to-reorder blocks

Legacy has `draggable-block-plugin.tsx` (grab a block by a handle, drop it elsewhere). The owned editor has no draggable / drop-indicator / move-by-drag — it can `remove-block` but not reposition by drag (`view/render/object-block.tsx:92-94`). The one direct-manipulation affordance legacy had that the owned editor dropped. The model already supports moving nodes (the table column/row move composes `move-node`); this is the view-layer handle + drop-indicator + a move command, not new core.

## 2. Editor overlay SPI — stop re-fixing React Aria popovers per call site

The recurring pain: every floating surface (selection flyout, cell `…` popover, ribbon `popover` actions, slash menu, link/annotation popovers) configures React Aria overlays raw, and each one re-derives the same focus/dismiss translation by hand — usually in raw-div land with string-matched `data-engine-*` selectors. We keep patching the same class of bug: dismiss-on-blur when focus bounces to the editor (`selection-flyout.tsx shouldKeepFlyoutOpen`, `table-interactions.tsx keepCellPopoverOpen` + `pressInsideRef`), don't-blur-the-editor-on-press (`slash-menu.tsx onMouseDown preventDefault`, `ribbon.tsx onMouseDownCapture`), autofocus-because-non-modal-won't (`use-reveal-focus.ts useAutoFocusWithin`), and the `@idco/ui` popover's `isNonModal` + `shouldCloseOnInteractOutside` passthrough. Five symptoms, one missing abstraction.

Root cause, stated once: React Aria overlays assume DOM focus is authoritative (focus leaves → dismiss; open → grab focus; close → restore). This editor assumes the opposite — the model selection is authoritative and DOM focus serves it (EditContext host, `focusSelectionSoon`, selection survives blur). Every fix is the same translation patch on that mismatch, written again at each site. The awfulness is not React Aria; it is using a focus-authoritative overlay library directly against a focus-non-authoritative surface, per call site. Wrap it, do not drop it — hand-rolling overlays would violate the UI philosophy and trade these bugs for worse a11y/positioning ones.

The idea: an editor-owned overlay adapter (an SPI in `packages/editor/src/view`, e.g. `view/overlay/`) that sits ON `@idco/ui`'s popover and encodes "focus is model-owned" once, so call sites declare intent, not React Aria mechanics. It must NOT live in `@idco/ui` — the "editing surface owns focus" semantics are editor-specific and the product-neutral package must not learn about EditContext / `view-root` (the boundary lint would reject it). `@idco/ui` stays the React-Aria-behavior + DaisyUI-styling layer; the editor adds a thin focus-ownership policy layer above it. This is the missing sibling to the command-surface / node-view / side-panel / data-provider SPIs: an overlay SPI.

Three responsibilities, and resist growing past them (this is a small core hook, not an overlay framework): (1) Surface-aware dismissal, inverted — the editor owns open/closed and React Aria becomes controlled and only *asks*; "RA never decides to close, it reports an interaction, the editor decides." The selection flyout already does this right (its coordinator owns it); the cell popover still leans on RA heuristics (the wrong model). Press-origin tracking + the surface-is-inside rule live here, defined against a real `surfaceRef`, not `closest("[data-engine-...]")`. (2) A declared focus policy — probably one enum is the whole thing: `form` (takes focus, autofocuses a field — glossary/comment/link) vs `actions` (focus-transparent, editor keeps focus, roving/keyboard — slash menu, flyout toolbar, cell `…` menu). That distinction subsumes `useAutoFocusWithin`, the `preventDefault` guards, and the blur-dismiss patches, because the primitive then knows whether a focus move *to the editor* is expected (always, for `actions`) or a real dismissal. (3) One portal root — portal every editor overlay into the editor root instead of `<body>`, so DOM ancestry matches React ancestry; containment checks become trivial and reliable, and the ribbon `onMouseDownCapture` hack (which exists only because React portals to body while synthetic events bubble the React tree) could likely be deleted.

Call-site shape should read as intent, e.g. `<EditorOverlay intent="actions" anchor={anchorRef} isOpen={open} onDismiss={…}>` — no `shouldCloseOnInteractOutside`, no `isNonModal`, no `pressInsideRef`, no surface selectors at the call site. First real design decision when planning: #1 (invert dismissal ownership) — get that contract right and the focus policy + portal fall out of it. Migration is incremental and self-justifying: move the flyout first (already ~80% this model), then cell popover, ribbon popovers, slash menu, link/annotation popovers — each migration deletes a pile of ad-hoc guards, so it pays for itself as you go. The subtle part is `intent: actions` (focus-transparent yet keyboard-operable); the slash menu already proves it is doable, the SPI just makes it the default for menus.

Origin: surfaced while fixing two non-modal-popover bugs (cell color popover dismissing on swatch press / not dismissing on outside click; toolbar-opened glossary/comment inputs unable to mouse-select because the toolbar's capture-phase `preventDefault` reached them through the React tree). Those are now patched, but the patches are the evidence this abstraction is missing.
