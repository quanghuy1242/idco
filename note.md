# Owned editor — parity gaps to close (grounded against legacy Lexical + the reader plan)

## What this is

A backlog of what the owned-model editor (`packages/editor`) still misses versus the legacy Lexical editor (`packages/editor-legacy`) and the read tier (docs/015), found by grounding the two side by side. Each item names the gap, where it is specified, the current state, and — for the SPI items — the concept it should be built on, so it can be picked up one at a time. This is a working list, not a design doc; it carries enough concept to start each slice without re-deriving it. Layout-shell parity (field label, bordered frame, error slot) and the side/aside TOC rail are deliberately out of scope; the aside rail was removed by design and the shell is the host's concern.

## 1. Drag-to-reorder blocks

Legacy has `draggable-block-plugin.tsx` (grab a block by a handle, drop it elsewhere). The owned editor has no draggable / drop-indicator / move-by-drag — it can `remove-block` but not reposition by drag (`view/render/object-block.tsx:92-94`). The one direct-manipulation affordance legacy had that the owned editor dropped. The model already supports moving nodes (the table column/row move composes `move-node`); this is the view-layer handle + drop-indicator + a move command, not new core.

## 2. Editor overlay SPI — stop re-fixing React Aria popovers per call site

The recurring pain: every floating surface (selection flyout, cell `…` popover, ribbon `popover` actions, slash menu, link/annotation popovers) configures React Aria overlays raw, and each one re-derives the same focus/dismiss translation by hand — usually in raw-div land with string-matched `data-engine-*` selectors. We keep patching the same class of bug: dismiss-on-blur when focus bounces to the editor (`selection-flyout.tsx shouldKeepFlyoutOpen`, `table-interactions.tsx keepCellPopoverOpen` + `pressInsideRef`), don't-blur-the-editor-on-press (`slash-menu.tsx onMouseDown preventDefault`, `ribbon.tsx onMouseDownCapture`), autofocus-because-non-modal-won't (`use-reveal-focus.ts useAutoFocusWithin`), and the `@idco/ui` popover's `isNonModal` + `shouldCloseOnInteractOutside` passthrough. Five symptoms, one missing abstraction.

Root cause, stated once: React Aria overlays assume DOM focus is authoritative (focus leaves → dismiss; open → grab focus; close → restore). This editor assumes the opposite — the model selection is authoritative and DOM focus serves it (EditContext host, `focusSelectionSoon`, selection survives blur). Every fix is the same translation patch on that mismatch, written again at each site. The awfulness is not React Aria; it is using a focus-authoritative overlay library directly against a focus-non-authoritative surface, per call site. Wrap it, do not drop it — hand-rolling overlays would violate the UI philosophy and trade these bugs for worse a11y/positioning ones.

The idea: an editor-owned overlay adapter (an SPI in `packages/editor/src/view`, e.g. `view/overlay/`) that sits ON `@idco/ui`'s popover and encodes "focus is model-owned" once, so call sites declare intent, not React Aria mechanics. It must NOT live in `@idco/ui` — the "editing surface owns focus" semantics are editor-specific and the product-neutral package must not learn about EditContext / `view-root` (the boundary lint would reject it). `@idco/ui` stays the React-Aria-behavior + DaisyUI-styling layer; the editor adds a thin focus-ownership policy layer above it. This is the missing sibling to the command-surface / node-view / side-panel / data-provider SPIs: an overlay SPI.

Three responsibilities, and resist growing past them (this is a small core hook, not an overlay framework): (1) Surface-aware dismissal, inverted — the editor owns open/closed and React Aria becomes controlled and only *asks*; "RA never decides to close, it reports an interaction, the editor decides." The selection flyout already does this right (its coordinator owns it); the cell popover still leans on RA heuristics (the wrong model). Press-origin tracking + the surface-is-inside rule live here, defined against a real `surfaceRef`, not `closest("[data-engine-...]")`. (2) A declared focus policy — probably one enum is the whole thing: `form` (takes focus, autofocuses a field — glossary/comment/link) vs `actions` (focus-transparent, editor keeps focus, roving/keyboard — slash menu, flyout toolbar, cell `…` menu). That distinction subsumes `useAutoFocusWithin`, the `preventDefault` guards, and the blur-dismiss patches, because the primitive then knows whether a focus move *to the editor* is expected (always, for `actions`) or a real dismissal. (3) One portal root — portal every editor overlay into the editor root instead of `<body>`, so DOM ancestry matches React ancestry; containment checks become trivial and reliable, and the ribbon `onMouseDownCapture` hack (which exists only because React portals to body while synthetic events bubble the React tree) could likely be deleted.

Call-site shape should read as intent, e.g. `<EditorOverlay intent="actions" anchor={anchorRef} isOpen={open} onDismiss={…}>` — no `shouldCloseOnInteractOutside`, no `isNonModal`, no `pressInsideRef`, no surface selectors at the call site. First real design decision when planning: #1 (invert dismissal ownership) — get that contract right and the focus policy + portal fall out of it. Migration is incremental and self-justifying: move the flyout first (already ~80% this model), then cell popover, ribbon popovers, slash menu, link/annotation popovers — each migration deletes a pile of ad-hoc guards, so it pays for itself as you go. The subtle part is `intent: actions` (focus-transparent yet keyboard-operable); the slash menu already proves it is doable, the SPI just makes it the default for menus.

Origin: surfaced while fixing two non-modal-popover bugs (cell color popover dismissing on swatch press / not dismissing on outside click; toolbar-opened glossary/comment inputs unable to mouse-select because the toolbar's capture-phase `preventDefault` reached them through the React tree). Those are now patched, but the patches are the evidence this abstraction is missing.

## 3. Duplication the `check:dup` gate is built not to notice

The fear is "vibe-coded `as any` / `isRecord` everywhere." Measured, that fear is misaimed: `as any` appears once across editor/reader/ui, `@ts-ignore`/`@ts-expect-error` zero — the type discipline is actually tight. The real debt is the opposite shape: the same *logic* maintained in parallel across packages, which is exactly what a duplicate gate should catch and this one structurally cannot. `scripts/check-duplication-threshold.mjs` is a repo-wide percentage gate (fail only above 3% mild-mode density); the repo sits at 1.9% (1713 duplicated lines, 37 clone groups), green. A percentage gate measures aggregate density, not per-block badness, so a single 350-line verbatim mirror never moves the needle and sails through. The semantic detector that *does* surface conceptual dupes (`pnpm advise:dupes`, fallow `--mode semantic`) exists but is not wired into `pnpm check`, so nothing fails on it.

Cluster A — the editor↔reader model mirror (`packages/editor/src/core/model` ↔ `packages/reader/src/reader/model`). `segmentText` is near-verbatim duplicated: same boundary-set construction, same sticky-merge branch, same sort, only the type names differ (`ResolvedMark`→`ReaderResolvedMark`). `marks.ts:74-115` ≡ `reader/model.ts:195-229` (337 tokens), plus model.ts↔model.ts, url-safety, and mark-render pairs, and the parity tests are themselves duplicated (`tests/editor/reader-resting-parity.test.tsx` ↔ `tests/reader.test.tsx`). This is the documented [[reader-convergence-028]] decision — resolution mirrored-and-parity-tested rather than relocated — so it is intentional, but it is still live duplication that drifts silently if one copy is edited and the parity test does not cover the edited path. Action is not "dedupe blindly"; it is "make the mirror a *declared* exception" so the gate acknowledges it instead of passing in ignorance.

Cluster B — two parallel rich-text renderers (`packages/reader/src/l1/*` ↔ `packages/ui/src/rich-text-content.tsx`), nine clone groups. `verticalAlignClass` and `readableTextColor` are literally copy-pasted — the reader copy's own comment reads "Moved verbatim from `@idco/ui` rich-text-content" — alongside table rendering (169+147 tokens), toc, objects, and types. This is the "reader / retire content-renderer" backlog item ([[editor-spi-gap-epic]]) quantified: ~900 tokens of a renderer that is supposed to go away but is being maintained twice in the meantime. Retiring `rich-text-content.tsx` in favor of the reader path collapses this cluster wholesale; until then it is the single largest avoidable dup surface in the tree.

Cluster C — `isRecord`, the micro-dup no token detector can see. Defined identically twice (`packages/editor/src/core/registry/object-registry.ts:595`, `packages/lib/src/auth-fetch.ts:160`) and reimplemented inline as a `typeof x === "object"` guard in ~13 files across editor/reader/lib. Each guard is ~6 tokens, below fallow's 50-token floor, so it is invisible to *any* token-based detector — the only thing that would catch it is a lint rule ("no inline record guard; import the shared one"), which does not exist. This is the class the user actually feels, and it is a lint problem, not a dup-gate problem.

Recommendations, smallest-leverage first. (1) Wire `advise:dupes` (semantic mode) into `pnpm check` as an allowlist gate: any *new* cross-package clone group fails; the two known-intentional mirrors (Cluster A model + its parity tests) sit in an explicit allowlist file, so the deliberate exception is recorded and a *third* mirror trips the wire. This converts "passes because the percent is low" into "passes because each dup is named." (2) Add one architecture-lint rule banning inline record guards, pointing at a single shared `isRecord` (promote one of the two definitions into `packages/lib` and import it) — this is the only thing that catches Cluster C at all. (3) Treat Cluster B as the existing retire-content-renderer backlog item, not new work; closing that item deletes nine clone groups for free. (4) Separately, `pnpm advise:aislop` already flags 7 errors worth a triage pass independent of duplication: a possible secret in `stories/engine-owned-model.stories.tsx:36`, two `dangerouslySetInnerHTML` sites (`reader/l1/blocks.tsx:217`, `ui/code-editor.tsx:135`), and `hono`/`undici` advisories. The throughline: the gate that exists rewards aggregate cleanliness; the duplication that hurts is concentrated and cross-package, so the fix is per-block accountability (allowlist + one lint rule), not a lower percentage threshold.

**Resolution (2026-06-25) — recommendations (1), (2), and the dedupable parts of (3) shipped.** (1) `scripts/check-semantic-dupes.mjs` runs the exact `advise:dupes` semantic detection and holds it against `scripts/semantic-dupes-allowlist.json`; it is wired into `pnpm check` as `check:dup:semantic` (alongside, not replacing, the percentage gate). Each clone *family* is keyed by its file-set (stable across line moves) and carries a content fingerprint, so a new cross-package clone fails for lacking an entry, an allowlisted mirror that silently drifts fails on fingerprint mismatch, and an entry with no written reason fails — `node scripts/check-semantic-dupes.mjs --update` refreshes fingerprints and surfaces new families without ever blessing one reasonlessly. The six surviving families are each named with a reason: Cluster A's `segmentText` mirror (docs/028), the two engine↔editor-legacy reimplementations (perf-metrics + column geometry, intentional because engine core cannot import the Lexical legacy package), the editor-barrel↔nav-icons detector false positive, and Cluster B's two renderer mirrors. (2) `architecture/no-inline-record-guard` (in `scripts/oxlint-js-plugins/architecture.js`, wired in `.oxlintrc.json`) bans inline `typeof x === "object"` record guards in @idco package source; the canonical `isRecord` now lives in `packages/lib/src/guards.ts` and all three former definitions + ~12 inline guards across editor/editor-legacy/reader import it. (3 partial) The pure copy-pasted helpers `verticalAlignClass`/`readableTextColor` moved to `packages/lib/src/rich-text-style.ts`, deleting one Cluster B clone family (reader/types ↔ ui rich-text-content) outright; the irreducible JSX renderer mirrors (table/toc) stay allowlisted against the retire-content-renderer backlog, which the cross-RSC-boundary constraint (reader L1 may not import @idco/ui) keeps as genuinely separate work. **Still open:** retiring `rich-text-content.tsx` itself (closes the last two Cluster B families and is a consumer-spanning migration), and recommendation (4)'s aislop security triage, which the note already brackets as independent of duplication.

## 4. Markdown / lists / export / publication hardening (2026-06-25 audit)

A grounded backlog across five areas the author asked about. The underlying *nodes* mostly already exist; these gaps are in the input, transport, and output layers around them. Block nodes present today: `heading` (h1–h6 via `tag`), `paragraph`, `listitem`, `quote`, `callout`, `code-block`, `media`, `embed`, `toc`, `table`, `post-ref`, `divider` (HR) — `core/registry/object-registry.ts:144-335`, `core/model/model.ts:200-215`. Inline marks present: `bold`, `italic`, `underline`, `strikethrough`, `code`, `highlight`, `subscript`, `superscript`, `link` — `core/model/model.ts:172-184`. The document snapshot already carries a top-level `settings: DocumentSettings` that round-trips through compat (`model.ts:265,285`), but it is an untyped `JsonObject`. So most items below are about *reaching* existing nodes (by typing, paste, export) and *typing* the settings bag — not new block types. **Today's focus = §4.1, §4.2, §4.3. §4.4 export and §4.5 page settings are deferred.** Calibration up front: not all of these are "easy" — the truly trivial ones (h4–h6 prefixes, persist list indent) sit beside two items that carry a real decision (the markdown-paste parser is a from-scratch parser; true list nesting is the synthesize-vs-structural fork in §4.3a). The rest is mechanical once those two calls are made.

### 4.1 Markdown typing affordances are shallow (`core/markdown-shortcuts.ts`) — TODAY

Current state: the detector is single-shot. Block prefixes fire only when the caret sits exactly after the prefix at offset 0 (`markdown-shortcuts.ts:117-129`); the only inline affordances are inline-code via a closing backtick, smart quotes, and bracket auto-pairing (`markdown-shortcuts.ts:142-162`). The view runs it once per single-character input in `view/render/text-block.tsx:174-180`. No multi-character paired-marker detection, no fence handling, no snippet/cascade engine.

Missing block prefixes: headings h4–h6 (`#### `…, only h1–h3 exist at `:88-90`; trivial, `heading` already takes any `tag`); task/checklist `- [ ] ` / `- [x] ` (the `listitem` `checked` attr exists at `model.ts:315` but nothing creates it — see §4.3c); horizontal rule `---`/`***`/`___` on its own line → insert a `divider` object (exists, `object-registry.ts:324`); code fence ` ``` ` → open a `code-block` object (exists, `:335`) capturing the language token. The HR and fence cases are *line→object* shortcuts — a shape the current detector (which only retypes a `TextLeafType`) does not model; they need a new `BlockShortcut` variant compiling to an object insert, and the fence additionally redirects subsequent input into the new block (the biggest block item). Also: only `1. ` is recognized; an arbitrary ordered start (`2. `, `7. `) needs a `value`/start attr or the adjacency-based ordinal paint (docs/018 §2.10) silently renumbers it.

Missing inline (paired-marker) affordances: bold `**`/`__`, italic `*`/`_`, strikethrough `~~`, highlight `==`, inline link `[text](url)`, autolink of a bare URL — all the marks exist and are reachable only by shortcut/toolbar today. Architecture play: inline-code already does the "scan back from the just-typed closing marker to a matching opener on the same line" move (`:144-150`); generalize *that one mechanism* into a paired-marker table (`marker → mark kind`, multi-char open/close, min-run guard) rather than six bespoke branches — one detector covers bold/italic/strike/highlight/code/link. Restraint applies (`[[spi-generalization-restraint]]`): this is the small core hook, not an autoformat plugin system, and explicitly **not** a multi-keystroke cascade/snippet engine (high-risk on the live-EditContext path, no concrete need — record that as a closed decision so it stops resurfacing).

### 4.2 Markdown paste + clipboard fidelity — TODAY

No markdown-on-paste: `use-clipboard.ts:117-120` reads `text/plain` and inserts it verbatim via `insert-text`; `text/markdown` is never read; the §4.1 block prefixes only fire on live single-char input so they do not help paste (`text-block.tsx:174`). Pasting a `.md` document drops all structure. Proposed: a pure markdown→snapshot-fragment parser (block + inline) that paste routes `text/markdown` (and optionally heuristically-markdown `text/plain`) through, producing the same node fragment the HTML-paste path builds, inserted as a multi-block paste. This is a from-scratch parser — the least "easy" item here; keep it pure/testable like the detector, and scope it to the representable node set rather than full CommonMark on day one.

HTML-paste gaps (`view/paste-html.ts`): paste maps headings/paragraphs/quote/list-items/code/links + inline formats, but drops tables (not in `BLOCK_TAGS` → degrade to cell text), images/`figure`/`figcaption` (not imported), and list nesting (flattened). Closing these reaches nodes that already exist (`table`, `media`) and shares the fragment builder with the §4.2 markdown parser — build it once. Clipboard image/file paste: `use-clipboard.ts` never inspects `clipboardData` for `image/*`; only drag-drop of images works (`owned-model-editor.tsx:180-207`) — add clipboard-image paste reusing that drop→`uploadImage`→`media` path. Copy/cut write `text/plain` only (`use-clipboard.ts:62,76`, via `collectSelectionText` in `core/store/selection.ts:61-101`), so editor→editor paste downgrades to plain text — also write `text/html` and (once §4.4 lands) `text/markdown`, and prefer the editor's own payload on internal paste. That last part depends on §4.4's serializers; do export first or share its block-walker.

### 4.3 Lists: nesting, persistence, checklist, styles — TODAY (`view/nodes/list.tsx`, `core/commands/blocks.ts`)

Lists are flat-by-design: `listitem` is a `TextLeafType` and depth is an `attrs.indent` integer rendered as left margin, not structural containment (docs/018 §2.10; `flat-blocks.ts:256`). A real structural `list` node and a `compileIndentItem` nesting path exist but are dead-by-design (`blocks.ts:374-457`, annotated UNREACHABLE).

(a) **True sublists vs. flat indent — a decision, not just code.** Today indent is purely visual; an indented bullet is still a ROOT child, so semantic `<ul><ol>` nesting, nested-list ordinal numbering, and markdown/HTML export of nested lists are all impossible from flat indent. Two roads: (i) keep flat leaves and *synthesize* the nesting tree at render/serialize time from the `indent` attr (cheaper, preserves per-item virtualization, but every consumer reconstructs the tree), or (ii) revive the structural `list` path (truthful tree, but reopens the recursive sub-block windowing the flat design deliberately deferred). This is the first real design decision in this section and gates 4.3b/4.3d and the list parts of §4.2/§4.4. Recommend (i) synthesize-from-indent first — unblocks export and HTML paste without reopening virtualization — and treat (ii) as the eventual multi-block-container work. (b) **Indent is dropped on save — a real data-loss bug, not just a gap.** `attrs.indent` is editor-session-only and is not carried through the compat export (docs/018 §2.8), so even the visual nesting an author makes today vanishes on reload. Highest-value, lowest-risk item: persist `indent` through compat import/export; independent of the 4.3a decision. (c) **Checklist has no UI.** The `checked` attr exists (`model.ts:315`) and legacy had full checklist UI, but import maps check lists to bullet (`payload-import.ts:188`) and there is no command/markdown to create or toggle one — add the `- [ ] ` prefix (§4.1), a toolbar/slash command, and a clickable checkbox in `list.tsx`. (d) **No glyph/numbering-style variants.** Bullets are a hardcoded `•`, ordered lists a hardcoded decimal (`list.tsx:26-42`, `view/styles.ts`); no model field for disc/circle/square/dash or 1./a./i./A./I. — needs a `listStyle` attr, render mapping, serialize support. Lowest priority; do after a–c.

### 4.4 Export: markdown + custom syntax — LATER (new; nothing exists)

No `toMarkdown`/`toHTML`/any export serializer exists in editor or reader; outputs are the native `EditorDocumentSnapshot` JSON and the legacy `RichTextCompatDocument` round-trip shape (`compat/compat.ts:198-217`). Proposed shape: a pure snapshot→text serializer walking `body.order`/`blocks`, with a per-object-kind `serialize(node) → string` seam on the node definition (`object-registry.ts`) so each object owns its custom syntax — callouts as `:::note` directives, code/mermaid as fenced blocks, divider as `---`, media as image syntax. Two hard constraints from docs/006: objects export their **baked static** representation only, never a live/computed one (006 §5.8); and markdown export must compose with the §4.2 parser so editor→md→editor is lossless for representable nodes, with a documented lossy set. Define the custom directive grammar once, here, as the export contract. Also unblocks the §4.2 copy-as-markdown fidelity item.

### 4.5 Page / publication settings — LATER (docs/006 §6)

The contract slot exists — `settings: DocumentSettings` round-trips (`model.ts:265,285`; `compat.ts:178,215`) — but it is an untyped `JsonObject`: no publication schema, no UI surface, no renderer parity. docs/006 §6.2 sketches the target `RichTextPublicationSettings` (page size, orientation, margins, header/footer, page numbers) and §6.5 the body-vs-settings boundary (global layout = settings; manual page-break = body node). Three pieces in order: (a) a *typed* `settings.publication` shape under the existing open bag — additive, the bag already passes unknown fields through (`compat.ts:176-180`); (b) a View-tab/object-chrome surface to edit it (006 §4.5) on React Aria + DaisyUI; (c) renderer parity in `@idco/reader` honoring the same settings with a host override (006 §6.3). Open upstream decision (006 §6.4): reflowable vs fixed-layout output determines how much of this is load-bearing vs advisory — settle with the host before building the fixed-layout pagination stage (its own large workstream). Editor's obligation regardless: preserve settings it accepts, never strip them.

Cross-cutting for all of §4: every markdown/list/export feature must round-trip editor↔reader↔export and carry a parity test (`[[reader-convergence-028]]`), and none of it pulls product/runtime deps into the editor (shared-package boundary).
